# Валидация соответствия проекта исходной идее

Дата проверки: 2026-03-30

Проверка выполнена по:
- `.memory-bank/project-idea.md`
- текущему коду репозитория
- `README.md`, `docs/release-readiness.md`, `Makefile`
- локальным прогонам `go test ./...`, `go vet ./...`, `golangci-lint run`

## Итог

Проект **частично соответствует** исходной идее, но в текущем состоянии есть несколько существенных расхождений и проблемных мест.

## Подтверждено как соответствующее

- Есть клиент-серверная архитектура с CLI-клиентом и сервером.
- Реализованы регистрация, аутентификация, CRUD для секретов, sync, работа с бинарными данными и банковскими картами.
- Есть поддержка версии и даты сборки CLI (`pkg/buildinfo/info.go`).
- Есть тесты, `go test ./...` и `go vet ./...` локально проходят.

## Несоответствия и проблемные места

### 1. CLI не даёт пользователю сохранять метаинформацию для записей

Это прямое расхождение с ТЗ: для любых данных должна быть возможность хранить произвольную текстовую метаинформацию (`.memory-bank/project-idea.md:35`).

Факты:
- В модели записи поле `Metadata` есть, но CLI только печатает его, а не запрашивает и не принимает от пользователя: `cmd/client/cli/main.go:198-199`.
- Команды `add` и `update` принимают только `type`, `name` и `data`, без отдельного ввода/флага для metadata: `cmd/client/cli/main.go:55-72`, `cmd/client/cli/main.go:262-291`, `cmd/client/cli/main.go:465-520`.

Следствие:
- Сервер и модель данных metadata поддерживают, но пользователь в основном MVP-клиенте не может ею пользоваться.

### 2. CLI по умолчанию не совпадает с TLS-only сервером и с quick start из README

Это одновременно функциональная проблема и несоответствие заявленной безопасной схеме.

Факты:
- Сервер поднимает gRPC только с TLS: `internal/app/server.go:286-313`.
- gRPC-клиент CLI при пустом `TLSCertFile` уходит в `insecure.NewCredentials()`: `pkg/apiclient/grpc/client.go:33-36`, `pkg/apiclient/grpc/client.go:43-51`.
- CLI при создании клиента не передаёт `TLSCertFile` вообще: `cmd/client/cli/main.go:80-86`.
- CLI по умолчанию стучится в `localhost:4400`: `cmd/client/cli/main.go:135-139`.
- README для сервера указывает gRPC-адрес `:9090` и предлагает запускать CLI без отдельной настройки TLS/cert/path/address: `README.md:82-84`, `README.md:107-114`.

Следствие:
- Из коробки CLI и сервер не согласованы по умолчанию ни по порту, ни по TLS-конфигурации.
- Quick start в README в текущем виде неоднозначен и, вероятнее всего, не воспроизводится без ручной дополнительной настройки.

### 3. Сервер может стартовать в режиме заглушек вместо явного отказа

Это опасное поведение для системы хранения приватных данных.

Факты:
- При пустом DSN сервер не падает, а запускается с `app.NewStubDeps()`: `cmd/server/main.go:53-56`.
- То же происходит при недоступной БД: `cmd/server/main.go:64-67`.
- То же происходит при ошибке миграций: `cmd/server/main.go:69-71`.

Следствие:
- Приложение может выглядеть как успешно запущенное, но фактически не реализовывать бизнес-функции хранения/синхронизации.
- Для password manager это слишком рискованное поведение и источник трудноуловимых ошибок эксплуатации.

### 4. Требование ТЗ про 70% покрытия по “коду всей системы” не подтверждено

Это прямой риск относительно `.memory-bank/project-idea.md:53-54`.

Факты:
- В `Makefile` порог считается не по всей системе, а после исключения части пакетов и файлов: `Makefile:9-25`.
- В `docs/release-readiness.md` заявлено “покрытие >= 70%”: `docs/release-readiness.md:11`.
- Доступные в репозитории coverage-артефакты этому противоречат:
  - `coverage.out` даёт `61.8%`
  - `coverage_clean.out` даёт `57.6%`
- Свежий `make cover-check` во время этой проверки пришлось запускать отдельно из-за ограничений sandbox; итоговую цифру команда в пределах сессии не вернула, поэтому надёжного подтверждения 70% по всей системе у меня нет.

Следствие:
- На текущий момент утверждение “ТЗ по покрытию выполнено” считаю **неподтверждённым**, а с учётом имеющихся coverage-артефактов скорее **ложным**.

## Неточности и спорные утверждения в документации

### 5. `docs/release-readiness.md` выглядит излишне оптимистичным

Факты:
- Документ утверждает “Готово к выпуску MVP” и что все критерии выполнены: `docs/release-readiness.md:5`, `docs/release-readiness.md:52-54`.
- При этом выше подтверждены реальные проблемы с metadata в CLI, с настройкой TLS/порта для CLI и со стартом сервера в stub-режиме.
- Локальный `golangci-lint run` у меня завершился ошибкой:
  - `Running error: context loading failed: no go files to analyze: running go mod tidy may solve the problem`

Следствие:
- Документ release readiness нельзя считать надёжным источником фактического состояния проекта без дополнительной перепроверки.

## Общий вывод

Если сравнивать проект именно с исходной идеей, то базовый password manager MVP в репозитории **в целом собран**, но до полного соответствия не хватает как минимум:
- полноценной работы с metadata в CLI;
- согласованной и рабочей TLS/gRPC-конфигурации CLI по умолчанию;
- отказа от запуска сервера в stub-режиме вместо fail-fast;
- прозрачного и подтверждённого выполнения требования по покрытию тестами не менее 70%.
