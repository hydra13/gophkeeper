# Task 21. Fail-fast bootstrap сервера без stub-режима

## Статус
Выполнено.

## Цель
Убрать опасный запуск сервера в режиме заглушек и перевести bootstrap в fail-fast модель, где отсутствие БД, ошибки миграций и критичной инфраструктуры приводят к явному завершению процесса.

## Описание
Для password manager особенно опасно состояние, когда сервер выглядит как успешно стартовавший, но фактически работает на заглушках и не обеспечивает надёжное хранение и синхронизацию данных. Сейчас такие ветки в bootstrap существуют для пустого DSN, недоступной БД и неуспешных миграций. Задача должна сделать поведение предсказуемым: при критической проблеме инфраструктуры процесс завершается с понятной ошибкой, логами и кодом возврата, а документация и тесты явно фиксируют такое поведение.

## Последовательность шагов
1. Найти все ветки bootstrap, которые переводят сервер на `stub deps` или иным образом маскируют критическую ошибку инфраструктуры.
2. Определить список обязательных зависимостей для production/dev server startup и оформить их как fail-fast требования.
3. Изменить `cmd/server` и связанный bootstrap так, чтобы пустой DSN, ошибка подключения к БД или ошибка миграций приводили к завершению старта с понятным сообщением.
4. Убедиться, что health/readiness не могут показывать ложноположительный результат при неинициализированном persistence слое.
5. Обновить runbook/README, если сценарий запуска теперь требует более явной предварительной подготовки инфраструктуры.
6. Добавить тесты на bootstrap и негативные сценарии старта.

## Критерии приемки
- [x] Сервер не стартует в режиме заглушек, если не задан обязательный DSN или другая критичная инфраструктурная зависимость.
- [x] При ошибке подключения к БД процесс старта завершается с ненулевым кодом и диагностическим сообщением, из которого понятно, что произошло.
- [x] При ошибке применения миграций сервер не продолжает работу и не поднимает transport-слои как будто сервис готов.
- [x] В коде bootstrap отсутствует путь, при котором критическая ошибка persistence маскируется рабочей конфигурацией с `stub deps`.
- [x] Документация по запуску сервера отражает fail-fast поведение и не обещает автоматический fallback в деградированный режим.
- [x] Добавлены автоматические тесты или bootstrap-проверки на пустой DSN, отказ БД и сбой миграций.
- [x] Поведение health/readiness согласовано с новым bootstrap и не сообщает о готовности сервиса, если критичные зависимости не инициализированы.
