# Task 12. Use-case слой записей и gRPC/HTTP интеграция

## Статус
Выполнена и перепроверена.

## Цель
Реализовать общий use-case слой для CRUD-операций над секретами и подключить его к gRPC services и HTTP endpoint-пакетам.

## Описание
Система обязана хранить четыре типа данных и метаинформацию для них. Эта задача покрывает создание, чтение, обновление, удаление и валидацию записей, но не включает sync и детальную механику binary uploads, чтобы объем был посилен для отдельной реализации. Для `binary`-записей здесь нужно работать только с метаданными и ссылкой на вложение; сам lifecycle чанков и скачивания остается в `task_13`.

## Последовательность шагов
1. Реализовать use-case слой для list/get/create/update/delete записей с учетом прав доступа и валидации типа.
2. Поддержать все обязательные типы данных: `login`, `text`, `binary`, `card`, включая metadata.
3. Для `binary`-типов поддержать связь записи с attachment/upload metadata без реализации transport upload-логики в этой задаче.
4. Реализовать soft delete и корректное скрытие удаленных записей в списках по умолчанию.
5. Подключить records use-case к HTTP records endpoints и gRPC records service.
6. Согласовать validation/error/conflict mapping между двумя transport-слоями.
7. Покрыть unit и integration-тестами CRUD-сценарии и ошибки валидации.

## Критерии приемки
- [x] Поддержаны все обязательные типы данных: `login`, `text`, `binary`, `card`.
- [x] Для любой записи можно хранить произвольную текстовую metadata.
- [x] Для `binary`-записей поддерживается связь с вложением без дублирования chunk upload-логики из `task_13`.
- [x] HTTP и gRPC используют один и тот же use-case слой и дают эквивалентное поведение.
- [x] Удаление реализовано через soft delete и учитывается в чтении/листинге.
- [x] Основные CRUD-сценарии проверены тестами.
