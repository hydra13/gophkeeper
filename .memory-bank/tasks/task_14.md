# Task 14. Sync use-case и разрешение конфликтов

## Цель
Построить механизм синхронизации между несколькими клиентами одного пользователя с учетом ревизий, pending-операций, soft delete и явного разрешения конфликтов.

## Описание
Синхронизация является обязательной частью исходной идеи проекта. По плану конфликт не должен разрешаться автомержем: сервер возвращает конфликт явно, а пользователь выбирает победившую версию. Эта задача должна закрыть серверную sync-логику и transport-интеграцию.

## Последовательность шагов
1. Зафиксировать серверную модель курсоров, ревизий, идентификаторов записей и правил изменения `updated_at`/`revision`.
2. Реализовать sync use-case для получения изменений с момента последнего курсора и отправки локальных изменений.
3. Подключить sync use-case к HTTP sync endpoints и gRPC sync service.
4. Реализовать обнаружение конфликта и структуру ответа сервера с локальной и серверной версиями записи.
5. Подготовить use-case для явного разрешения конфликта пользователем без автомержа.
6. Написать integration/e2e-тесты на одновременное изменение записи с двух устройств, soft delete и восстановление после сетевого сбоя.

## Критерии приемки
- [ ] Синхронизация работает инкрементально.
- [ ] Для каждой записи поддерживаются идентификатор и ревизия, достаточные для обнаружения конфликтов.
- [ ] Конфликт возвращается явно и не разрешается автоматически на сервере.
- [ ] HTTP и gRPC используют один и тот же sync use-case слой.
- [ ] Сценарии с несколькими устройствами покрыты тестами.
- [ ] Soft delete корректно участвует в синхронизации.
