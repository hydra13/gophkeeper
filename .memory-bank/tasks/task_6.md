# Task 6. HTTP records endpoints в `api/`

## Цель
Подготовить endpoint-first HTTP-контракты и каркас обработчиков для CRUD-операций над записями секретов.

## Описание
Хранение секретов является базовой функцией из исходной идеи проекта. На HTTP-слое это должно выражаться отдельными endpoint-пакетами для list/create/get/update/delete, а не общим handler-слоем. Контракты должны покрывать все обязательные типы записей и metadata.

## Последовательность шагов
1. Создать пакеты `api/records_v1_get`, `api/records_v1_post`, `api/records_by_id_v1_get`, `api/records_by_id_v1_put`, `api/records_by_id_v1_delete`.
2. В каждом пакете подготовить `handler.go`, `handler_test.go` и `mocks/`.
3. Описать DTO для `login`, `text`, `binary`, `card`, включая metadata и служебные поля.
4. Зафиксировать правила сериализации ревизий, soft delete и ответов о конфликте.
5. Согласовать HTTP-ошибки и коды ответов с моделью данных и gRPC-контрактами.
6. Добавить unit-тесты на валидацию, отсутствие доступа, несуществующую запись и конфликт ревизий.

## Критерии приемки
- [ ] Для CRUD по записям есть отдельные endpoint-first пакеты в `api/`.
- [ ] DTO покрывают все обязательные типы данных и metadata.
- [ ] В HTTP-контрактах отражены ревизии, soft delete и conflict response.
- [ ] Ошибки и коды ответов согласованы между endpoint-пакетами.
- [ ] На каждый endpoint есть базовый набор unit-тестов.
