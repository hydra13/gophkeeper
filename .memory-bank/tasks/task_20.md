# Task 20. Согласовать CLI с TLS-only сервером и воспроизводимым quick start

## Статус
Выполнено. Повторная проверка 2026-03-30 пройдена.

## Цель
Сделать так, чтобы CLI и сервер были согласованы по умолчанию по адресу, TLS-конфигурации и quick start-сценарию, а запуск из документации воспроизводился без неявных ручных донастроек.

## Описание
В текущем состоянии CLI по умолчанию использует другой адрес и может подключаться без TLS, тогда как сервер поднимается только с TLS. Из-за этого безопасная схема запуска, описанная в проектной идее и README, на практике оказывается неоднозначной и легко ломается. Задача должна выровнять значения по умолчанию, механизм передачи dev-сертификата и quick start документацию так, чтобы пользователь мог поднять локальное окружение и сразу пройти базовый сценарий регистрации, логина и работы с данными.

## Последовательность шагов
1. Зафиксировать целевой сценарий локального запуска: какие адреса, сертификаты и переменные окружения считаются canonical для dev/quick start.
2. Привести дефолтный адрес CLI в соответствие с фактическим адресом сервера или, если нужен иной адрес, явно синхронизировать оба значения через конфигурацию и документацию.
3. Убрать неявный insecure path для стандартного CLI-сценария и обеспечить загрузку TLS-certificate/CA для подключения к TLS-only серверу.
4. Сделать удобный dev flow: либо через dev certificate bundle в репозитории, либо через понятный путь генерации/передачи сертификата в CLI.
5. Перепроверить bootstrap/CLI команды из README и других runbook-файлов, чтобы quick start был полным и самодостаточным.
6. Добавить smoke/integration-покрытие на успешное TLS-подключение CLI к локальному серверу по documented path и на ожидаемую ошибку при некорректной TLS-конфигурации.

## Критерии приемки
- [x] CLI по умолчанию использует тот же gRPC endpoint, который документирован и действительно поднимает сервер в базовом локальном сценарии.
- [x] Основной путь подключения CLI к серверу работает только через TLS и не сваливается в `insecure` без явного осознанного режима разработки.
- [x] Пользователь может запустить quick start из документации без скрытых шагов и получить рабочий сценарий `register/login/add/list` с CLI.
- [x] Путь к сертификату или trust bundle для CLI либо настроен по умолчанию, либо однозначно документирован и обязателен в примерах запуска.
- [x] Ошибки TLS-конфигурации дают понятное сообщение и не маскируются под сетевые или доменные ошибки другого типа.
- [x] README, runbook'и и help-тексты CLI не содержат противоречий по портам, адресам, TLS-режиму и обязательным параметрам запуска.
- [x] Добавлены автоматические проверки, которые подтверждают успешное подключение CLI по TLS в documented-конфигурации и неуспешное подключение без корректного сертификата.
