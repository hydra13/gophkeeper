# Task 19. Поддержка metadata в CLI для всех типов записей

## Статус
Выполнено. Повторная проверка 2026-03-30 пройдена.

## Цель
Устранить расхождение с ТЗ и дать пользователю CLI полноценную возможность задавать, изменять и просматривать произвольную текстовую metadata для записей всех поддерживаемых типов.

## Описание
Сейчас доменная модель, transport и сервер поддерживают поле `Metadata`, но основной CLI-сценарий не позволяет пользователю передавать это поле при создании и обновлении записей. Из-за этого проект не соответствует исходной идее password manager, где любая запись должна хранить не только полезные данные, но и дополнительную текстовую метаинформацию. Задача должна закрыть разрыв на уровне UX, контрактов CLI и тестового покрытия, не ломая текущие сценарии работы с уже существующими типами записей.

## Последовательность шагов
1. Проанализировать текущие CLI-команды `add`, `update`, `list`, `get` и определить, где metadata должна передаваться флагом, отдельным аргументом или интерактивным вводом.
2. Добавить в CLI явный и единообразный способ передавать metadata для новых записей и для обновления существующих записей.
3. Обеспечить корректную обработку пустой metadata, очистки metadata и сохранения многострочного или длинного текстового значения без искажения.
4. Проверить, что metadata корректно проходит через `clientcore`/API-клиент к серверу и не теряется при sync, list и get.
5. Обновить help/usage и документацию CLI, чтобы пользователь понимал, как задавать и менять metadata.
6. Добавить unit/smoke/e2e-тесты, покрывающие сценарии создания, обновления, чтения и очистки metadata.

## Критерии приемки
- [x] Команда создания записи в CLI позволяет явно передать `metadata` для каждого поддерживаемого типа записи.
- [x] Команда обновления записи в CLI позволяет изменить существующую `metadata` без побочных изменений остальных полей.
- [x] Пользователь может явно очистить `metadata`, и после сохранения/получения запись возвращается с пустым значением без старых данных.
- [x] При `get` и других сценариях просмотра CLI отображает `metadata` предсказуемо и без потери форматирования.
- [x] Значение `metadata` доходит до сервера и возвращается обратно без потери данных в end-to-end сценарии.
- [x] Старые сценарии CLI без передачи `metadata` продолжают работать и не требуют обязательного ввода нового параметра.
- [x] Обновлены `--help`, README или другая пользовательская документация по CLI-командам, где появился новый способ работы с `metadata`.
- [x] Добавлены автоматические тесты, которые отдельно подтверждают создание записи с `metadata`, обновление `metadata`, очистку `metadata` и обратную совместимость сценария без `metadata`.
