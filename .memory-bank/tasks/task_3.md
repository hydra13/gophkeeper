# Task 3. Доменная модель, инварианты синхронизации и загрузок

## Цель
Зафиксировать единую доменную модель проекта: записи секретов, сессии, ревизии, конфликты, загрузки бинарных данных и версии ключей без зависимости от transport и storage деталей.

## Описание
Это центральная задача для всей реализации. По `.memory-bank/project-idea.md` система должна хранить `login`, `text`, `binary`, `card`, поддерживать metadata, синхронизацию между устройствами, а по `.memory-bank/plan.md` еще и `device_id`, `soft delete`, `revision`, `key_version`, `payload_version`, chunk uploads и явное разрешение конфликтов. Все дальнейшие слои должны опираться именно на эти инварианты.

## Последовательность шагов
1. Описать сущности `User`, `Record`, типизированные payload-модели для `login`, `text`, `binary`, `card`, а также `Session`, `RecordRevision`, `SyncConflict`, `UploadSession`, `KeyVersion`.
2. Зафиксировать обязательные поля и семантику: `id`, `user_id`, `type`, `metadata`, `revision`, `deleted_at`, `device_id`, `key_version`, `payload_version`, timestamps.
3. Определить инварианты изменения ревизий, soft delete и конфликтов синхронизации.
4. Определить доменную модель chunk upload/download: `upload_id`, `chunk_index`, статус загрузки, завершение, resume.
5. Описать доменные ошибки и правила валидации для некорректного типа записи, конфликта ревизий, истекшей сессии, неизвестной версии ключа и незавершенной загрузки.
6. Добавить документацию к экспортируемым типам и unit-тесты на ключевые инварианты.

## Критерии приемки
- [ ] Есть единая доменная модель для всех обязательных типов секретов.
- [ ] В домене формализованы ревизии, soft delete, конфликты и `device-aware` сессии.
- [ ] Для бинарных данных описан отдельный жизненный цикл upload/download с resume.
- [ ] Домен не зависит от HTTP, gRPC, PostgreSQL, файлового хранилища или CLI.
- [ ] Экспортируемые типы и ошибки документированы.
- [ ] Unit-тесты покрывают ключевые инварианты доменной модели.
