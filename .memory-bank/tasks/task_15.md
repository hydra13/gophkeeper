# Task 15. Shared client core и локальный кеш

## Цель
Выделить общий клиентский слой для CLI: API-клиент, auth flow, sync engine, локальный кеш и управление pending-операциями без привязки к конкретному UI.

## Описание
Клиент не должен напрямую работать с transport и локальным storage. Нужен `pkg/clientcore`, ориентированный на текущий CLI и пригодный для возможного будущего `desktop`, но без преждевременной ориентации на `web`, так как по плану для web нужен отдельный security-трек.

## Последовательность шагов
1. Спроектировать пакеты `pkg/apiclient`, `pkg/clientcore` и `pkg/cache` с четкими границами ответственности.
2. Реализовать transport-адаптеры клиента так, чтобы выбранный transport можно было заменить без переписывания use-case слоя клиента; при этом CLI MVP может использовать один transport по умолчанию.
3. Добавить в client core методы уровня приложения: `Login`, `ListRecords`, `GetRecord`, `SaveRecord`, `DeleteRecord`, `SyncNow`.
4. Реализовать локальный кеш метаданных, payload-ов, pending-операций и состояния незавершенных upload/download.
5. Поддержать офлайн-работу с повторной отправкой изменений после восстановления связи.
6. Покрыть shared client core unit-тестами и сценариями офлайн/повторной синхронизации.

## Критерии приемки
- [ ] CLI работает через `pkg/clientcore`, а не напрямую через transport или repositories.
- [ ] Локальный кеш поддерживает metadata, payload, pending-операции и upload/download state.
- [ ] Shared client core не зависит от конкретного UI.
- [ ] Слой ориентирован на CLI MVP и не тащит преждевременные web-specific решения.
- [ ] Есть тесты на офлайн-режим, повтор отправки и восстановление upload/download.
- [ ] Границы между `apiclient`, `clientcore` и `cache` документированы и соблюдаются кодом.
