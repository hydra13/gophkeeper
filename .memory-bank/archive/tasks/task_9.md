# Task 9. PostgreSQL, файловое хранилище и миграции

## Статус
Выполнена.

## Цель
Реализовать слой персистентности для пользователей, записей, сессий, ревизий, sync state, upload state и key versions на PostgreSQL, а для бинарных payload-ов подготовить файловое хранилище.

## Описание
По плану MVP сервер хранит метаданные в PostgreSQL и бинарные вложения отдельно. Эта задача должна закрепить схему БД, обратимые миграции, репозитории и транзакционные границы, на которые затем лягут auth, records, uploads и sync.

## Последовательность шагов
1. Спроектировать схему БД для пользователей, записей, payload metadata, ревизий, сессий, upload sessions, upload chunks и key versions.
2. Создать обратимые SQL-миграции в `migrations/` в формате `goose`, где каждый файл содержит секции `-- +goose Up` и `-- +goose Down`, и добавить механизм их применения.
3. Реализовать PostgreSQL-репозитории для пользователей, записей, сессий, sync state, uploads и key versions.
4. Реализовать файловое blob-хранилище для бинарных данных и связать его с upload session моделью.
5. Добавить транзакционные сценарии для операций, где метаданные и состояние загрузки должны меняться согласованно.
6. Написать integration-тесты для миграций и репозиториев, включая ограничения уникальности, soft delete и resume-state.

## Критерии приемки
- [x] В `migrations/` есть полный набор SQL-миграций для MVP-схемы в формате `goose`, где каждый файл содержит `Up` и `Down` секции.
- [x] Репозитории покрывают пользователей, записи, сессии, sync state, uploads и key versions.
- [x] Для бинарных payload-ов используется отдельное файловое хранилище.
- [x] Поддерживаются незавершенные upload-сессии и информация по чанкам.
- [x] Миграции и репозитории проверены integration-тестами.
- [x] Слой персистентности не протекает деталями SQL в доменный слой.
