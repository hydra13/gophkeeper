# Task Dependencies

## Как читать документ
- Секция `Summary` нужна человеку: в ней указан рекомендуемый порядок и логика группировки.
- Для автоматического разбора моделью используйте `.memory-bank/tasks/deps_ai.md`.
- Поле `depends_on` означает прямые зависимости.
- Поле `blocks` показывает задачи, которые рационально начинать после текущей.
- Поле `parallel_group` показывает рекомендуемую волну параллельного исполнения.
- Поле `mvp` показывает, входит ли задача в обязательный MVP.

### Уточнения по зависимостям
- `task_8` сознательно ограничена инфраструктурным bootstrap и не требует полной готовности persistence/use-case слоев; иначе она конфликтовала бы с порядком из `plan.md`.
- `task_11` зависит и от `task_5`, и от `task_4`, потому что auth должен подключаться одновременно к HTTP и gRPC transport-слоям.
- `task_12` покрывает `binary`-записи только на уровне record metadata и attachment reference; chunk lifecycle остается в `task_13`.

### Что можно выполнять параллельно
- После `task_3`: `task_4`, `task_5`, `task_6`, `task_7`
- После `task_11`: `task_12` и `task_13`
- После `task_15`: часть smoke/e2e сценариев можно готовить параллельно с финализацией `task_16`

### Что не входит в обязательный MVP
- Полноценная реализация `desktop` клиента
- Полноценная реализация `web` клиента
- OTP, TUI, import/export

### Что сознательно исключено из task backlog
- Отдельная задача на `cmd/client/desktop` и `cmd/client/web` как продуктовые клиенты не нужна: по плану это post-MVP направления.
- Создание пустых директорий и фиксация post-MVP границ должны быть закрыты внутри `task_1` и `task_18`, а не отдельной реализационной задачей.
