# План: переход на S3-совместимое blob-хранилище через MinIO

Дата: 2026-03-30

## Цель

Перевести серверное blob-хранилище бинарных данных с локальной файловой системы на S3-совместимое хранилище, используя `MinIO` в локальной разработке как Docker-контейнер, аналогично уже используемому `PostgreSQL`.

Итоговое поведение:

- в dev-окружении сервер работает с `MinIO`, поднятым локально в Docker;
- в коде сохраняется абстракция `repositories.BlobStorage`;
- выбор backend-а происходит через `blob.provider`;
- текущие upload/download сценарии продолжают работать без изменения внешнего API;
- документация и конфиги по умолчанию ведут разработчика к запуску `PostgreSQL + MinIO`.

## Текущее состояние

На момент планирования в проекте уже есть базовые предпосылки для перехода:

- в `internal/config/config.go` у `BlobStorageConfig` уже есть поля `provider`, `endpoint`, `bucket`, `access_key`, `secret_key`, `region`;
- `internal/config/loader.go` уже умеет читать эти поля из файла, env и флагов;
- `internal/config/validate.go` пока валидирует только обязательность `blob.path` для `provider=local`;
- `cmd/server/main.go` всегда создаёт `storage.NewLocalBlob(cfg.Blob.Path)`;
- текущая реализация blob-хранилища находится в `internal/storage/local.go`;
- `README.md` и `configs/config*.json` всё ещё описывают локальное файловое blob-хранилище как основной путь для разработки.

Это значит, что основная работа лежит не в конфиге, а в реализации нового storage backend-а, выборе провайдера при bootstrap и обновлении dev-инфраструктуры.

## Объём фичи

Фича включает:

- новый S3 backend для реализации `repositories.BlobStorage`;
- поддержку `provider=s3` в bootstrap сервера;
- локальный `MinIO` для разработки через Docker;
- настройку bucket-а и dev-конфига;
- тесты на новый backend и выбор провайдера;
- обновление документации и quick start.

Фича не включает:

- миграцию уже существующих локальных blob-файлов автоматически при старте сервера;
- замену протокола upload/download;
- изменение схемы БД;
- presigned URLs, multipart upload на стороне S3 или прямую загрузку из клиента в MinIO.

## Предлагаемое решение

### 1. Оставить контракт хранения без изменений

Слой `repositories.BlobStorage` уже достаточно абстрактный:

- `Save(path string, data []byte) error`
- `Read(path string) ([]byte, error)`
- `Delete(path string) error`
- `Exists(path string) (bool, error)`

Его стоит сохранить без изменений, чтобы:

- не трогать бизнес-логику upload/download;
- не менять PostgreSQL-репозиторий и upload-сервисы сверх необходимости;
- ограничить фичу инфраструктурным слоем хранения.

### 2. Добавить отдельную реализацию S3 storage

Новый backend логично разместить рядом с текущим storage-слоем, например:

- `internal/storage/s3.go`

или отдельным пакетом:

- `internal/storage/s3/`

Предпочтительный подход: отдельный пакет или отдельный файл в `internal/storage`, но с тем же интерфейсом `BlobStorage`.

В реализации нужно:

- принимать `endpoint`, `bucket`, `access_key`, `secret_key`, `region`;
- уметь работать с S3-compatible endpoint MinIO;
- использовать path как object key без изменения контракта вызывающего кода;
- корректно обрабатывать `not found` для `Read`, `Delete`, `Exists`;
- по возможности не читать лишние метаданные, если для `Exists` достаточно `StatObject`/`HeadObject`.

### 3. Выбор backend-а вынести в фабрику

Вместо прямого вызова `storage.NewLocalBlob(cfg.Blob.Path)` нужен единый конструктор, например:

- `storage.NewBlobStorage(cfg.Blob)`

Эта фабрика должна:

- при `provider=local` создавать текущий файловый backend;
- при `provider=s3` создавать новый S3 backend;
- при неизвестном provider возвращать fail-fast ошибку;
- инкапсулировать различия инициализации и базовую проверку доступности backend-а.

Это позволит не разносить логику выбора storage по приложению.

## План реализации

## Этап 1. Подготовить storage abstraction для нескольких provider-ов

### 1.1 Ввести единый конструктор blob storage

Изменить bootstrap так, чтобы `cmd/server/main.go` не знал о конкретной реализации.

Изменения:

- добавить фабрику выбора provider-а в `internal/storage`;
- заменить прямой вызов `storage.NewLocalBlob(...)` на вызов фабрики.

Результат:

- точка выбора backend-а одна;
- дальнейшее расширение storage-провайдеров остаётся локализованным.

### 1.2 Нормализовать ошибки storage-слоя

Нужно заранее решить, как код различает:

- объект отсутствует;
- backend недоступен;
- ошибка конфигурации.

Минимальный вариант:

- сохранить текущее поведение интерфейса;
- внутри `Read/Delete/Exists` приводить ошибки S3/MinIO к ожидаемой семантике.

Это особенно важно для уже существующих сценариев delete-if-exists и чтения чанков.

## Этап 2. Реализовать S3 backend поверх MinIO-compatible API

### 2.1 Выбрать клиентскую библиотеку

Нужно выбрать один из двух рабочих путей:

- AWS SDK for Go v2;
- официальный `minio-go`.

Предпочтение:

- `AWS SDK v2`, если нужен более нейтральный S3-совместимый слой и потенциальный выход в настоящий S3 без смены API;
- `minio-go`, если важнее минимализм и простой старт именно под MinIO.

Рекомендуемое решение для проекта: `AWS SDK for Go v2`, потому что фича формулируется как переход на `S3`, а `MinIO` выступает dev-реализацией.

### 2.2 Реализовать операции Save/Read/Delete/Exists

Новый backend должен:

- `Save` -> `PutObject`;
- `Read` -> `GetObject` с полным чтением тела;
- `Delete` -> `DeleteObject` с идемпотентным поведением;
- `Exists` -> `HeadObject`.

Дополнительно:

- для MinIO понадобятся `endpoint` и режим path-style addressing;
- соединение должно поддерживать статические креды из конфига;
- в dev-режиме нужно явно поддержать `http`, без требования TLS до MinIO.

### 2.3 Добавить защиту от некорректных object key

Хотя S3 не требует файловой семантики, path по-прежнему приходит из прикладного кода.

Нужно сохранить базовую дисциплину:

- не принимать пустой path;
- не позволять неожиданные абсолютные пути и path traversal как смысловую ошибку;
- нормализовать key одинаково для local и s3 backend-ов.

Это снизит риск расхождений между provider-ами.

## Этап 3. Расширить конфигурацию и валидацию под `provider=s3`

### 3.1 Уточнить поддерживаемые значения `blob.provider`

Явно зафиксировать поддерживаемые provider-ы:

- `local`
- `s3`

При любом другом значении сервер должен завершаться с понятной ошибкой конфигурации.

### 3.2 Добавить обязательные поля для `provider=s3`

В `internal/config/validate.go` нужно валидировать:

- `blob.endpoint` обязателен;
- `blob.bucket` обязателен;
- `blob.access_key` обязателен;
- `blob.secret_key` обязателен;
- `blob.region` либо обязателен, либо заполняется дефолтом.

Рекомендуемый компромисс:

- для явности требовать `region` в конфиге;
- для dev-конфига использовать фиксированное значение вроде `us-east-1`.

### 3.3 При необходимости добавить дополнительные поля

Возможно, текущего конфига окажется недостаточно. Потенциально полезные поля:

- `blob.use_path_style` для S3-compatible endpoint-ов;
- `blob.create_bucket` для dev bootstrap;
- `blob.secure` или вывод этого флага из `endpoint`.

Чтобы не раздувать конфиг без нужды, лучше начать без новых полей, если:

- path-style можно жёстко включить для `provider=s3`;
- create bucket будет делаться на уровне dev-инфраструктуры, а не серверного bootstrap;
- `http/https` можно корректно определить по `endpoint`.

Если в коде всплывёт неоднозначность, тогда уже добавить ровно одно поле: `blob.use_path_style`.

## Этап 4. Поднять MinIO как часть локальной разработки

### 4.1 Добавить docker-сценарий запуска MinIO

Сейчас quick start описывает запуск `PostgreSQL` отдельной командой `docker run`.

Нужно выбрать один из вариантов:

- добавить ещё одну команду `docker run` для MinIO;
- перейти на `docker compose` для `postgres + minio`.

Предпочтительный вариант: `docker compose`, потому что после появления второй обязательной зависимости локальная разработка становится существенно удобнее в одном файле.

Минимальный состав compose:

- `postgres`;
- `minio`;
- опционально `mc`/init-container для создания bucket-а.

### 4.2 Автоматически создавать bucket для dev

Критично избежать ручного шага "зайди в UI MinIO и создай bucket".

Нужно сделать один из вариантов:

- init container с `mc mb`, который создаёт bucket при старте;
- make-команду для инициализации окружения;
- серверный bootstrap с create-if-not-exists для dev.

Предпочтительный вариант: init container или отдельный one-shot service в `docker compose`, чтобы dev-окружение поднималось воспроизводимо без скрытых ручных действий.

### 4.3 Обновить dev-конфиг по умолчанию

`configs/config.dev.json` должен быть переключён на `provider=s3`, например:

- `provider: "s3"`
- `endpoint: "http://localhost:9000"`
- `bucket: "gophkeeper-dev"`
- `access_key` / `secret_key` из dev-значений
- `region: "us-east-1"`

При этом `blob.path` можно оставить пустым или убрать из dev-конфига.

## Этап 5. Обновить bootstrap и поведение fail-fast

### 5.1 Проверять доступность blob backend-а на старте

Сервер уже работает в fail-fast режиме для БД и миграций. Для storage нужно сохранить ту же модель.

При старте сервера ошибки должны всплывать сразу, если:

- невалидная конфигурация S3;
- bucket не существует;
- MinIO недоступен;
- креды не подходят.

Это важно, чтобы сервер не стартовал с "живой" БД и "мертвым" blob backend-ом.

### 5.2 Зафиксировать поведение для bucket provisioning

Нужно принять одно правило:

- либо сервер только подключается к уже существующему bucket;
- либо сервер создаёт bucket при старте.

Рекомендуемое решение:

- в приложении считать bucket внешней зависимостью и не создавать его автоматически;
- создание bucket-а закрыть dev compose/init слоем.

Так поведение будет безопаснее для production и предсказуемее для эксплуатации.

## Этап 6. Покрыть изменение тестами

### 6.1 Юнит-тесты для S3 backend-а

Нужно добавить тесты минимум на:

- успешный `Save/Read`;
- `Exists` для существующего и отсутствующего объекта;
- `Delete` для существующего и отсутствующего объекта;
- ошибки инициализации при пустом endpoint/bucket;
- нормализацию и валидацию key/path.

Если полноценный MinIO в unit-тестах неудобен, допустимы:

- таблица тестов на внутреннюю нормализацию ключей;
- интеграционные тесты с MinIO отдельно.

### 6.2 Тесты конфигурации

Расширить `internal/config/validate_test.go`:

- успешный `provider=s3` при полном наборе полей;
- ошибки при пустых `endpoint/bucket/access_key/secret_key/region`;
- ошибка при неизвестном provider.

Текущий тест `BlobPathNotRequiredForNonLocalProvider` стоит уточнить до строгого сценария `provider=s3`.

### 6.3 Тесты bootstrap/wiring

Нужно добавить покрытие для `cmd/server/main.go` или вынесенной фабрики:

- выбор `local` backend-а;
- выбор `s3` backend-а;
- fail-fast при неизвестном provider;
- fail-fast при ошибке создания blob storage.

### 6.4 Интеграционные тесты с MinIO

Отдельный полезный уровень проверки:

- поднять MinIO в тестовом окружении;
- выполнить `Save -> Exists -> Read -> Delete`.

Если полноценная автоматизация сразу дорогая, минимально можно ограничиться:

- smoke-сценарием в документации;
- backlog-пунктом на автоматизацию интеграционных storage-тестов.

Но желательный вариант для этой фичи: хотя бы один автоматизированный integration test с реальным S3-compatible backend-ом.

## Этап 7. Обновить документацию и developer experience

### 7.1 README

Нужно обновить:

- MVP/архитектурное описание: вместо "локальное файловое хранилище" указать `S3-compatible blob storage`, а для dev явно назвать `MinIO`;
- quick start: запуск `PostgreSQL + MinIO`;
- таблицу конфигурации: описать `blob.provider`, `blob.endpoint`, `blob.bucket`, `blob.access_key`, `blob.secret_key`, `blob.region`;
- smoke-сценарий: добавить ожидания по поднятию MinIO.

### 7.2 Примеры конфигурации

Обновить:

- `configs/config.dev.json` под MinIO;
- `configs/config.example.json` так, чтобы он показывал оба поддерживаемых сценария или хотя бы production-friendly `s3`.

Если один файл не может удобно показать оба варианта, стоит:

- оставить `config.example.json` как `s3`;
- при необходимости добавить отдельный `config.localblob.example.json` только как fallback/dev-reference.

### 7.3 Makefile и команды запуска

Если будет введён `docker compose`, полезно добавить удобные цели:

- `make dev-up`
- `make dev-down`
- `make dev-reset`

Это не строго обязательно для фичи, но сильно снижает трение для локального запуска.

## Риски и решения

### Риск 1. Разная семантика local FS и S3

Проблема:

- файловая система и object storage по-разному ведут себя с путями, атомарностью и отсутствующими объектами.

Снижение риска:

- держать единый контракт `BlobStorage`;
- явно тестировать семантику `Exists/Delete/Read not found`;
- нормализовать object key в одном месте.

### Риск 2. Случайная привязка реализации именно к MinIO

Проблема:

- можно реализовать код так, что он будет работать только с MinIO, но не с обычным S3-compatible API.

Снижение риска:

- выбирать нейтральный S3 API;
- в документации называть MinIO dev-реализацией, а не единственным провайдером;
- не завязываться на MinIO-специфичные API без необходимости.

### Риск 3. Непрозрачный dev bootstrap

Проблема:

- если bucket создаётся вручную, разработка быстро становится хрупкой.

Снижение риска:

- создавать bucket автоматически через compose init-container;
- описать один воспроизводимый путь запуска в README.

### Риск 4. Регрессия существующих upload/download сценариев

Проблема:

- storage меняется инфраструктурно, но ломает существующие сценарии загрузки бинарных файлов.

Снижение риска:

- не менять интерфейс `BlobStorage`;
- сохранить path/key формат без изменения БД и API;
- прогнать существующие upload-related тесты и добавить storage smoke.

## Критерии готовности

Фича считается завершённой, если выполнено всё ниже:

- сервер умеет стартовать с `blob.provider=local` и `blob.provider=s3`;
- при `provider=s3` сервер успешно работает с MinIO;
- локальная разработка поднимается через воспроизводимый Docker-сценарий;
- bucket для dev создаётся автоматически;
- `configs/config.dev.json` по умолчанию настроен на MinIO;
- `README.md` отражает новый способ запуска;
- есть тесты на конфиг, bootstrap и новый storage backend;
- существующие upload/download сценарии не требуют изменения клиентского API.

## Рекомендуемый порядок выполнения

1. Добавить фабрику выбора storage provider-а и покрыть её тестами.
2. Реализовать S3 backend и юнит-тесты на его семантику.
3. Расширить валидацию конфига для `provider=s3`.
4. Подключить новый backend в `cmd/server/main.go`.
5. Поднять `MinIO` в Docker и автоматическое создание bucket-а.
6. Переключить `configs/config.dev.json` на `s3`.
7. Обновить `README.md`, Makefile и smoke-инструкции.
8. Прогнать регрессию существующих upload/download и server bootstrap тестов.

## Ожидаемые точки изменений в коде

- `cmd/server/main.go`
- `internal/storage/local.go`
- `internal/storage/*` с новой S3-реализацией и фабрикой
- `internal/config/validate.go`
- `internal/config/validate_test.go`
- `internal/config/loader_test.go` при необходимости уточнения сценариев
- `configs/config.dev.json`
- `configs/config.example.json`
- `README.md`
- `Makefile`
- Docker/compose-файлы для локальной инфраструктуры

## Решения, которые стоит принять до начала реализации

1. Используем ли `docker compose` как основной dev bootstrap вместо отдельных `docker run`.
2. Какую библиотеку берём для S3-клиента: `AWS SDK v2` или `minio-go`.
3. Нужен ли отдельный конфиг-флаг вроде `blob.use_path_style`, или достаточно зашить это поведение для `provider=s3`.
4. Должен ли сервер только подключаться к уже существующему bucket, или разрешаем create-if-not-exists.

Рекомендуемые ответы для этой кодовой базы:

- `docker compose` — да;
- клиент — `AWS SDK v2`;
- отдельный `use_path_style` — только если без него нельзя сделать читаемо;
- создание bucket-а — не в приложении, а в dev bootstrap.
