# План реализации TUI-клиента для GophKeeper

## Цель

Сделать отдельный TUI-клиент на базе `tview`, который работает с тем же сервером и тем же `ClientCore`, что и текущий CLI.

Поведение на старте:
- показать 2 действия: `login` и `register`;
- добавить явную кнопку `exit`;
- также оставить возможность выйти через прерывание процесса;
- после успешного `register` показать статус регистрации и кнопку `OK`;
- по нажатию `OK` автоматически выполнять login с теми же `email/password`;
- рабочая сессия доступна только после `login`.

Поведение после логина:
- пользователь получает доступ к тем же сценариям, что и в CLI: `list`, `get`, `add`, `update`, `delete`, `sync`, `logout`.

## Опорные точки в текущем коде

- Текущий CLI уже построен вокруг общего слоя [`pkg/clientcore/core.go`](/Users/rasabirov/sources/_my/gophkeeper/pkg/clientcore/core.go).
- Команды CLI собраны в [`cmd/client/cli/commands`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/cli/commands) и уже содержат полезную прикладную логику: разбор типов записей, формат payload, загрузку/скачивание бинарных данных, sync и logout.
- Bootstrap клиента и восстановление авторизации находятся в [`cmd/client/cli/bridge.go`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/cli/bridge.go).
- `Register` в `ClientCore` сейчас не создаёт авторизованную сессию, поэтому для UX `register -> continue as logged in` лучше не менять core-контракт, а последовательно вызывать `Register`, затем `Login` с теми же credentials на уровне TUI orchestration.

## Предлагаемая структура

### 1. Новый entrypoint

Добавить новый клиент:
- [`cmd/client/tui/main.go`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/tui/main.go)

Задачи:
- инициализация `tview.Application`;
- создание `ClientCore` через общий bootstrap;
- запуск root-screen;
- централизованная обработка ошибок и graceful cleanup.

### 2. Общий bootstrap для CLI и TUI

Чтобы не дублировать подключение к серверу, кешу и TLS-конфигу, стоит вынести код из [`cmd/client/cli/bridge.go`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/cli/bridge.go) в переиспользуемый пакет, например:
- [`cmd/client/common/bootstrap.go`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/common/bootstrap.go)

Туда перенести:
- `defaultNewCore`;
- вычисление `cache dir`;
- определение адреса сервера;
- поиск TLS-сертификата;
- `hostname`;
- восстановление auth через `core.RestoreAuth()`.

CLI после этого должен продолжить работать через тот же shared bootstrap.

### 3. TUI-слой

Собрать отдельный пакет, например:
- [`cmd/client/tui/app`](/Users/rasabirov/sources/_my/gophkeeper/cmd/client/tui/app)

Внутри разделить ответственность:
- `app.go` — жизненный цикл приложения;
- `state.go` — текущее состояние UI, выбранная запись, статус, ошибки;
- `screens_auth.go` — `login` и `register`;
- `screens_main.go` — основной layout после логина;
- `dialogs_record.go` — формы `add`, `update`, `get`, `delete`;
- `presenters.go` — форматирование записей, ошибок и статусов.

Это удержит TUI-код отдельно от `ClientCore` и не даст бизнес-логике утечь в UI.

## Экранная модель

### 1. Стартовый экран

Показать модальное меню с двумя действиями:
- `Login`
- `Register`
- `Exit`

Требования:
- `Exit` должен штатно завершать приложение;
- `Ctrl+C` тоже остаётся рабочим способом прервать клиент;
- фокус и навигация должны работать только клавиатурой.

### 2. Экран login

Форма:
- `email`
- `password`
- `Login`
- `Back`

Поведение:
- по `Login` вызывать `core.Login(...)`;
- при успехе переходить в основной экран;
- при ошибке показывать modal с текстом ошибки;
- после `Back` возвращаться на стартовый экран.

### 3. Экран register

Форма:
- `email`
- `password`
- `Register`
- `Back`

Поведение:
- по `Register` вызывать `core.Register(...)`;
- при успехе показывать modal `registered successfully`;
- в modal должна быть только кнопка `OK`;
- по `OK` запускать `core.Login(...)` с теми же `email/password`;
- при успешном автологине переводить пользователя в основной экран;
- если login после register не удался, показывать ошибку и переводить пользователя на экран `login` с предзаполненным `email`;
- `Back` возвращает на стартовый экран без сайд-эффектов.

### 4. Основной экран после login

Лучший стартовый MVP-layout:
- слева список записей;
- справа detail/status pane;
- сверху или снизу action bar с подсказками по клавишам и действиям.

Доступные действия:
- `list` — перезагрузить список и фильтр по типу;
- `get` — показать полную карточку записи;
- `add` — открыть форму создания;
- `update` — открыть форму редактирования выбранной записи;
- `delete` — запросить подтверждение и удалить запись;
- `sync` — запустить синхронизацию и показать статус;
- `logout` — выполнить `core.Logout(...)` и вернуть пользователя на стартовый экран.

## Маппинг CLI-сценариев в TUI

### 1. List

Источник данных:
- `core.ListRecords(ctx, recordType)`

Что показать в списке:
- `ID`
- `Type`
- `Name`
- `Revision`
- признак `deleted`
- краткий `metadata`

Дополнительно:
- фильтр по типу: `all | login | text | binary | card`;
- отдельный refresh через `list` или горячую клавишу.

### 2. Get

Для небинарных записей:
- вызывать `core.GetRecord(...)`;
- показывать detail-view рядом со списком или в modal.

Для бинарных записей:
- не пытаться рендерить содержимое в терминале;
- показывать метаданные и размер;
- отдельно предлагать сохранить файл по пути через input dialog;
- для сохранения вызывать `core.DownloadBinary(...)` и затем писать файл на диск.

### 3. Add

Сделать форму выбора типа записи:
- `login`
- `text`
- `binary`
- `card`

Дальше открывать специализированную форму:
- `login`: `name`, `metadata`, `login`, `password`;
- `text`: `name`, `metadata`, `content`;
- `card`: `name`, `metadata`, `number`, `holder`, `expiry`, `cvv`;
- `binary`: `name`, `metadata`, `file path`.

Поведение:
- собрать `models.Record`;
- вызывать `core.SaveRecord(...)`;
- для `binary` после сохранения вызывать `core.UploadBinary(...)`;
- после успеха обновлять список и показывать статус.

### 4. Update

Поведение:
- работать от выбранной записи в списке;
- заранее подставлять текущее значение полей;
- поддержать изменение `metadata`, включая очистку;
- для `binary` принимать новый `file path`, если пользователь хочет заменить содержимое;
- сохранять через `core.SaveRecord(...)`;
- при обновлении бинарного payload запускать `core.UploadBinary(...)`.

### 5. Delete

Поведение:
- удаление только для выбранной записи;
- сначала confirmation dialog;
- затем `core.DeleteRecord(...)`;
- после успеха обновить список и очистить detail-pane, если удалили текущую запись.

### 6. Sync

Поведение:
- отдельное действие `Sync now`;
- вызывать `core.SyncNow(...)`;
- показывать явный результат: `synced` или текст ошибки;
- после успеха перечитывать список записей.

### 7. Logout

Поведение:
- вызывать `core.Logout(...)`;
- очищать локальное UI-state;
- возвращать на стартовый экран `login/register`.

## Что лучше переиспользовать из CLI

Чтобы TUI не дублировал прикладные детали, стоит вынести из CLI в shared helpers:
- разбор и валидацию `RecordType`;
- построение `models.RecordPayload` по данным формы;
- форматирование краткой и полной записи;
- поиск записи по `id`/`name`, если такой сценарий понадобится в TUI;
- операции работы с бинарными файлами: чтение из пути и запись в путь.

Отдельно для auth-flow:
- вынести общий метод orchestration уровня TUI/controller, например `loginWithCredentials(email, password)`;
- после `Register` вызывать именно этот метод, а не пытаться программно "нажимать" UI-кнопку `Login`.

Подход:
- оставить CLI-команды как thin wrappers;
- общую прикладную логику поднять в пакет вида `pkg/clientui` или `cmd/client/common/uihelpers`.

## Основные риски и как их снизить

### 1. Дублирование логики между CLI и TUI

Риск:
- часть logic по payload и бинарным файлам сейчас сидит в CLI-командах.

Снижение:
- перед TUI вынести общие helper/use-case куски в shared-пакет.

### 2. Смешение UI и бизнес-логики

Риск:
- `tview`-обработчики быстро превращаются в длинные callback-цепочки.

Снижение:
- держать формы и модалки тонкими;
- всю сборку данных и orchestration выносить в контроллеры/презентеры.

### 3. Бинарные записи в терминале

Риск:
- TUI не должен пытаться отображать бинарный payload как текст.

Снижение:
- для binary показывать только метаданные;
- download/save делать отдельным явным действием.

### 4. Автологин после регистрации

Риск:
- если зашить логику прямо в обработчики кнопок, TUI быстро станет хрупким.

Снижение:
- не эмулировать нажатие кнопки `Login`;
- вынести общий auth-method и использовать его и из формы логина, и из post-register flow.

## Definition of Done

Задача считается завершённой, когда:
- существует отдельный бинарь TUI-клиента;
- на старте он показывает `login`, `register`, `exit`;
- `register` завершает сценарий success-экраном с `OK`, после которого выполняется автологин;
- после `login` доступны `list`, `get`, `add`, `update`, `delete`, `sync`, `logout`;
- TUI использует тот же `ClientCore`, что и CLI;
- CLI после выделения shared bootstrap остаётся рабочим;
- сценарии auth, CRUD, sync и logout покрыты тестами или как минимум воспроизводимыми smoke-check шагами.
