# Оптимизированный план реализации GophKeeper

## Принципы плана
- MVP строится вокруг `server + cli`, где CLI является обязательным клиентом первой поставки.
- `desktop` рассматривается как следующий приоритет после CLI, если останется время на реализацию и стабилизацию.
- `web` не входит в MVP и проектируется как отдельный следующий этап с собственной security-проработкой.
- Сервер поддерживает два равноправных transport слоя: `gRPC` и `HTTP REST`.
- Import/export данных в MVP не реализуется и остаётся отдельным post-MVP направлением.
- Все соединения между клиентом и сервером работают только через TLS.
- Основная защита данных в MVP обеспечивается на сервере: шифрование при хранении, `master-key + периодические data keys`, контролируемая ротация и фоновый re-encryption.
- На клиентской стороне допускается открытый кеш данных; безопасность клиентского окружения считается ответственностью пользователя.

## Этап 1. Архитектурный каркас и правила проекта

### 1.1 Зафиксировать MVP-границы
Подтвердить состав первой поставки: `server` и `cli`.
Отдельно пометить `desktop` как post-MVP+ приоритет 2, а `web` как отдельный следующий трек.

### 1.2 Утвердить модульную структуру репозитория
Согласовать структуру директорий для `cmd`, `internal`, `pkg`, `api`, `migrations`.
Shared-слои проектировать с прицелом на переиспользование в будущем `desktop`, но без усложнения CLI MVP.

### 1.3 Настроить инженерную обвязку
Подготовить `Makefile`, линтеры, генерацию protobuf, шаблоны конфигурации и единый формат запуска тестов.
Зафиксировать правила покрытия, документации и сборки артефактов.

## Этап 2. Домен и модель данных

### 2.1 Спроектировать доменные сущности
Описать модели пользователя, записи секрета, метаданных, ревизий, устройств и статусов синхронизации.
Базовая сущность хранения должна быть общей для `login`, `text`, `binary`, `card`, а сессии должны сразу поддерживать `device_id`.

### 2.2 Определить модель хранения и шифрования
Разделить пользовательские данные, служебные поля, версию серверного ключа и формат хранения бинарных payload.
Сразу заложить `key_version`, `payload_version`, а для бинарных объектов также `upload_id`, `chunk_index`, состояние загрузки и поддержку re-encryption.

### 2.3 Определить инварианты синхронизации
Зафиксировать идентификаторы записей, ревизии, soft delete, формат pending-операций и ответ при конфликте.
Для конфликтов принять явный выбор пользователя без авто-мержа.

## Этап 3. Контракты API и транспорт

### 3.1 Спроектировать gRPC и HTTP контракты
Описать gRPC-контракты и отдельно HTTP REST API для auth, records, sync, attachments и служебных операций.
Оба transport слоя считаются равноправными и должны покрывать один и тот же бизнес-функционал без расхождения поведения, включая chunk upload/download и resume-сценарии.

### 3.2 Зафиксировать правила синхронизации transport слоёв
Определить единые DTO, карту ошибок, правила версионирования и тестовую матрицу для gRPC и HTTP.
Нужно исключить ситуацию, когда один transport поддерживает сценарий, а второй нет.

### 3.3 Ввести версионирование и TLS-стратегию
Разнести proto по версиям (`rpc/proto/v1`) и утвердить правила совместимости.
Определить выпуск сертификатов для dev/prod и запретить работу по незашифрованному каналу.

## Этап 4. Серверная платформа и хранилища

### 4.1 Реализовать конфигурацию и bootstrap сервера
Поднять конфиг из флагов, env и файла, собрать DI для БД, file storage, crypto и transport.
Подготовить полноценный запуск gRPC-сервера и HTTP API, а также общий graceful shutdown.

### 4.2 Реализовать слой персистентности
Сделать репозитории для пользователей, записей, ревизий, сессий и серверных ключей.
Для MVP использовать PostgreSQL для метаданных и файловое хранилище для бинарных вложений.

### 4.3 Подготовить миграции и начальные данные
Создать миграции схемы БД, индексы, ограничения уникальности и таблицы для key rotation.
Сразу заложить обратимые миграции, таблицу версий ключей со статусами `active/deprecated/retired` и сценарий инициализации пустой системы.

## Этап 5. Криптография и управление ключами

### 5.1 Реализовать серверное шифрование данных
Подготовить сервис encrypt/decrypt для данных на стороне сервера и хранения `key_version`.
Пароли пользователей хранить только в виде безопасных хешей, без обратимого восстановления.

### 5.2 Реализовать key management
Ввести `master-key` сервера и периодические `data encryption keys` с версионированием.
Каждая запись должна быть связана с `key_version`, а в системе должна появиться таблица версий ключей со статусами `active`, `deprecated`, `retired`.

### 5.3 Реализовать rotation и re-encryption
Зафиксировать плановую ротацию ключей периода, например раз в `90 дней`, и поддержку внеплановой ротации при инциденте.
Реализовать фоновый batch-процесс перешифрования старых данных, который идемпотентен, перезапускаем после сбоя и логирует прогресс.

## Этап 6. Аутентификация, авторизация и сессии

### 6.1 Реализовать регистрацию по email и password
Добавить регистрацию пользователя по `email + password` с обязательной валидацией email.
Сразу определить формат user identity, уникальность email и правила нормализации адреса.

### 6.2 Реализовать логин и управление сессиями
Поднять безопасный логин, выпуск access/refresh токенов и хранение сессий по устройствам.
Нужно предусмотреть `device-aware sessions`: `device_id`, имя устройства, тип клиента, `last_seen_at`, отзыв сессии и истечение срока действия.

### 6.3 Подключить middleware/interceptors
Реализовать auth, rate limit и нормализованную карту ошибок для gRPC/HTTP.
Эта же задача закрывает базовую защиту от brute-force и выравнивает поведение transport слоёв.

## Этап 7. Основной сервис записей

### 7.1 Реализовать CRUD для секретов
Поднять use-case слой создания, чтения, обновления и удаления записей без привязки к транспорту.
Валидация должна зависеть от типа записи, но хранение оставаться единообразным.

### 7.2 Реализовать работу с бинарными вложениями
Добавить загрузку, чтение и удаление бинарных payload-ов через абстракцию storage.
Сразу реализовать multipart/chunk upload с `upload session`, возобновлением после обрыва и управлением большими файлами как частью основного MVP.

### 7.3 Реализовать базовое логирование
Логировать как минимум входящие запросы и исходящие ответы.
Дополнительную observability оставить расширяемой задачей после MVP.

## Этап 8. Синхронизация и оффлайн-режим

### 8.1 Определить протокол синхронизации
Зафиксировать формат `pull changes`, `push pending`, server conflict response и курсоры синхронизации.
Для MVP оставить on-demand sync с ручным разрешением конфликтов пользователем.

### 8.2 Реализовать локальный кеш клиента
Сделать локальное хранилище метаданных, кеш payload-ов и очередь pending-операций.
Кеш в MVP допускается хранить в открытом виде на стороне клиента, но для бинарных вложений нужно поддержать незавершённые upload/download с возможностью продолжения.

### 8.3 Реализовать сценарии конфликта
Добавить обнаружение конфликта, сохранение локальной версии и запрос явного выбора пользователю.
Автоматическое слияние не делать.

## Этап 9. Общая клиентская библиотека

### 9.1 Выделить shared client core
Собрать в одном модуле API-клиент, auth flow, sync engine, cache manager и единый use-case слой.
Этот модуль должен быть ориентирован на CLI сейчас и пригоден для будущего `desktop`.

### 9.2 Подготовить единый use-case API
Описать команды уровня приложения: `Login`, `ListRecords`, `GetRecord`, `SaveRecord`, `SyncNow`.
UI-слои не должны напрямую работать с transport или локальным хранилищем.

### 9.3 Реализовать device-aware sessions
Закрепить в shared-модели и серверных контрактах `device_id`, имя устройства, тип клиента и `last_seen_at`.
Эта модель нужна для отзыва доступа по устройству, аудита и прозрачной многоклиентской синхронизации.

## Этап 10. CLI-клиент

### 10.1 Реализовать пользовательские команды
Сделать `register`, `login`, `list`, `get`, `add`, `update`, `delete`, `sync`, `logout`.
Команды должны работать через shared client core, а не отдельную бизнес-логику.
CLI-клиент должен жить в `cmd/client/cli`, а не в отдельном корневом `cmd/cli`.

### 10.2 Добавить удобство и безопасность UX
Поддержать ввод чувствительных значений без эха, форматированный вывод и подтверждение опасных операций.
Сюда же входит вывод версии и даты сборки бинарника.

### 10.3 Покрыть CLI сценарии тестами
Проверить happy-path, offline режим, конфликты, ошибки сети, истечение сессии и восстановление загрузки больших бинарных файлов после обрыва.
CLI остаётся основным референсным клиентом для отладки всей системы.

## Этап 11. Desktop-клиент (при наличии времени)

### 11.1 Подготовить каркас desktop приложения
Настроить Wails-приложение и адаптеры к shared client core.
Этап стартует только после стабилизации серверного API и CLI.
Desktop-клиент должен жить в `cmd/client/desktop`.

### 11.2 Реализовать основные пользовательские сценарии
Добавить auth, список записей, просмотр, редактирование, sync status и работу с вложениями.
Desktop является вторым приоритетом после готового CLI MVP.

### 11.3 Подготовить сборку под целевые ОС
Настроить reproducible build, версионирование, упаковку и базовые smoke-тесты для macOS, Linux и Windows.
Этот этап имеет смысл только при достаточном запасе времени.

## Этап 12. Web-клиент (post-MVP)

### 12.1 Уточнить security-модель web
Проработать смешанный режим аутентификации и безопасное хранение сессионного материала в браузере.
До этой проработки web не включать в основной scope реализации.

### 12.2 Спроектировать browser-specific адаптеры
Определить границы shared-логики и то, что останется на стороне web shell.
Отдельно описать ограничения для IndexedDB/local storage и XSS/CSRF модели.

### 12.3 Реализовать базовые экраны
Добавить auth, список записей, просмотр, редактирование, удаление и экран конфликта.
Эта работа должна стартовать только после закрытия web security policy.
Web-клиент должен жить в `cmd/client/web`.

## Этап 13. Тестирование и качество

### 13.1 Покрыть домен и сервисы unit-тестами
Тестами закрыть доменную логику, sync engine, repositories и use-case слой.
Цель: стабильная база для достижения покрытия 70%+ без искусственного добивания процентов.

### 13.2 Добавить интеграционные тесты
Проверить gRPC API, HTTP API, миграции, TLS-подключение, файловое хранилище и key rotation.
Отдельно прогнать end-to-end сценарии регистрации, CRUD, sync, re-encryption, смены `key_version` и chunked upload/download с resume.

### 13.3 Настроить CI pipeline
Добавить проверки `fmt`, `lint`, `test`, `cover`, генерацию proto и smoke-сборку CLI.
Дополнительные проверки desktop/web добавлять по мере появления этих клиентов в scope.

## Этап 14. Документация и выпуск MVP

### 14.1 Описать архитектуру и эксплуатацию
Подготовить README, схему модулей, описание TLS, key rotation, sync и локального кеша.
Документация должна отражать реальный MVP scope: `server + cli`, выбранную схему `master-key + периодические data keys`, а также явно фиксировать отсутствие import/export в первой поставке.

### 14.2 Подготовить policy-документы
Поддерживать эксплуатационную документацию по ротации ключей, расписанию перевыпуска и batch re-encryption.
Эти правила должны быть частью основной архитектурной документации, а не отдельным открытым policy-вопросом.

### 14.3 Провести release readiness review
Проверить покрытие, документацию, минимальную безопасность, сборку CLI и незакрытые риски.
После этого можно фиксировать scope MVP и переносить остальное в backlog.

- Этапы 1-10, 13, 14.

### Желательно после MVP
- Этап 11.

### Отдельный следующий трек
- Этап 12.
