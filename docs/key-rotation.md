# Ротация ключей шифрования

## Обзор

GophKeeper использует двухуровневую модель шифрования (envelope encryption):
- **Мастер-ключ** (`crypto.master_key` в конфигурации) — 32-байтный ключ, хранится вне БД
- **Data keys** — уникальные 32-байтные ключи для каждой версии, шифруются мастер-ключом и хранятся в таблице `key_versions`

Жизненный цикл data key:
```
active → deprecated → retired
```

- **active** — используется для шифрования новых данных и расшифровки
- **deprecated** — не используется для шифрования, но допускает расшифровку (пока данные не перешифрованы)
- **retired** — ключ выведен из обращения, все данные перешифрованы на новую версию

## Процедура ротации ключа

Ротация выполняется через `keys.Manager.Rotate()`:
1. Текущий active ключ переводится в статус **deprecated**
2. Создаётся новая версия data key в статусе **active**
3. Все новые записи шифруются уже новым active ключом

Фоновый job `reencrypt.Job` автоматически находит записи, зашифрованные deprecated-ключами, и перешифровывает их текущим active-ключом.

## Ручная ротация (эксплуатационный сценарий)

Ротация ключей инициируется вызовом метода `Rotate()` менеджера ключей (`keys.Manager`). В рамках MVP автоматического административного endpoint или CLI-команды для ротации нет — процедура требует прямого вызова из кода сервера.

**Порядок действий оператора:**

1. Убедитесь, что сервер запущен и подключение к БД работает
2. Инициируйте ротацию одним из способов:
   - Добавьте временный вызов `keys.Manager.Rotate()` в код сервера и перезапустите его
   - Или вызовите `Rotate()` через отдельную admin-команду (если добавлена в CLI/entrypoint)
3. Дождитесь завершения фонового перешифрования (job `reencrypt`)
4. После завершения перешифрования вызовите `keys.Manager.Retire(version)` для перевода deprecated-ключей в статус `retired`

> **Ограничение MVP:** ротация ключей не автоматизирована и не имеет выделенного admin endpoint. Процедура выполняется вручную через вызов из кода. Автоматизация запланирована в post-MVP.

## Параметры фонового job перешифрования

| Параметр        | Значение по умолчанию | Описание                           |
| --------------- | --------------------- | ----------------------------------- |
| `batchSize`     | 100                   | Количество записей за одну итерацию |
| `interval`      | 30s                   | Интервал между запусками            |

## Обработка инцидентов

### Компрометация мастер-ключа

1. Немедленно сгенерируйте новый 32-байтный мастер-ключ
2. Обновите `crypto.master_key` в конфигурации
3. Перезапустите сервер — при старте будет создана новая версия data key
4. Запустите перешифрование всех данных
5. После завершения переведите старые ключи в retired

### Невозможно расшифровать данные

1. Проверьте, что `crypto.master_key` в конфигурации соответствует ключу, которым были зашифрованы data keys
2. Проверьте статус ключа в `key_versions` — ключ должен быть `active` или `deprecated`
3. Проверьте целостность `encrypted_key` и `key_nonce` в `key_versions`
4. Если мастер-ключ утерян — восстановление данных невозможно

### Re-encryption job не завершается

1. Проверьте логи сервера на ошибки при обработке батчей
2. Проверьте, что в `key_versions` есть активная версия
3. Проверьте подключение к БД и blob-хранилищу
4. При необходимости увеличьте `batchSize` или `interval` через опции job
